# 博客大纲（固定结构）

目标读者：懂深度学习，但不熟悉该论文所在垂直领域。写作要做到“像同事聊天一样讲清楚”，同时保证每个结论都可追溯到论文或代码证据。

## 0) 标题与元信息块（开头必须有）

```
【文献阅读】| <模型/方法名>：<一句话主线>
> 论文标题：<来自 paper_meta.json / paper_text>
> 作者：<作者列表>
> 期刊/会议：<venue>
> 发布时间：<YYYY-MM-DD>
> 影响因子（JCR 2024）：<数值（Q1/Q2/Q3/Q4）>
> 中科院分区（2025）：<1/2/3/4（Top：是/否）>
> DOI：<链接>
```

指标获取规则：
- 影响因子/分区/Top/Open Access 优先用本 skill 内的本地库查询（`references/journal_metrics_2025.sqlite3`），不要凭记忆或网上随手搜。
- 查询脚本：`scripts/query_journal_metrics.py --journal "<venue>"`
- 查不到就写“未披露/未收录”，不要脑补。

紧接着插入封面裁剪图：

`【图1：论文首页信息】![](paper/front_matter/page1_header.png)`

## 1) 5 分钟 TL;DR（先把主线钉死）

3-5 条要点，覆盖：
- 任务是什么/为什么难
- 方法亮点（1-2 句直觉 + 1 句落地）
- 实验最硬的结论（带关键数字）
- 代码怎么跑通（最短闭环）
- 论文 vs 代码差异（如有）

## 2) 任务与痛点

把任务讲具体：
- 输入/输出/约束/成功标准（尽量对应论文定义）
- 当前方法的典型 trade-off（不要写成泛泛综述）

## 3) 方法主线（讲故事，只留 3-4 个转折）

每个转折写清楚三件事：
1) 引入了什么表示/模块/训练目标  
2) 为什么能解决某个痛点  
3) 代码里大概怎么实现（描述思路即可，不要堆路径清单）

插图规则（强制）：
- 只能用 `paper/figures/` 与 `paper/front_matter/` 的裁剪图，不允许用 `paper/pages/` 整页图。

## 4) 训练与实验（用证据说话）

建议拆成 3-4 小节：
- 训练在学什么（少量公式/伪代码可以，但要克制）
- 指标解释（挑 2-3 个关键指标讲清楚）
- 关键结果（挑 2-3 组对比，写“对谁/看哪张图/一句话结论”）
- 消融/边界（可选但建议有）

## 5) 代码落地地图（可复现 + 可改）

最少包含：
- 最短复现路径（环境/数据/命令/输出）
- 5-10 个最关键开关（步数/阈值/采样/过滤/配置等）
- 想做研究改动，先从哪里动手（安全改动 vs 激进改动）

## 6) 学习者视角：我带走什么？

3-6 条“可迁移的脑中模型”，每条标注来源（论文/代码）。

## 7) 差异与空白清单（强烈推荐）

把“论文没写清/代码默认值/实现细节差异”列出来，并明确写“无法从当前输入确认”的部分。

## 8) 参考文献（末尾必须有）

- `[1]` 必须是本次论文（只用输入里可核验的信息）。
- 其他条目只收录输入中明确出现的 DOI/arXiv/URL。

## 9) 图床上传与回填（完稿后的最后一步）

当正文里图片还在用本地相对路径时，运行：

`scripts/sync_lsky_images.py --workspace <项目目录> --markdown <blog.md>`

它会上传缺失图片到 Lsky，并把 Markdown 里的本地图片链接替换为 URL，同时更新 `.lsky_upload_cache.json`。

## 10) 自检与润色（交付前）

交付前再走一遍硬性检查与可读性润色：

- `scripts/check_article_requirements.py <blog.md> --require-urls`
- `scripts/check_article_length.py <blog.md> --min 5000 --max 7000`
- 对照 `references/self_review.md` 做最后一轮“讲明白/讲清楚”润色
